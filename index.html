model Order {
  fields {
    customer_id: id       required
    amount:      currency  positive
    status:      OrderStatus
  }

  create {
    before(body) {
      body.amount > 0
      body.amount <= 10000
    }
    after(body) {
      Notify(customer_id: body.customer_id  event: "order_created")
    }
  }

  read {}
  update {}
  delete {}
}
A different approach

Model your domain.
The rest follows.

Most backend frameworks give you tools and leave the structure up to you. Fookie takes the opposite stance — the structure is the framework.

One file describes everything

Fields, validation rules, lifecycle hooks, background jobs, external integrations — all in one schema. A new developer reads the .fql file and knows exactly what the system does. No hunting across controllers, services, and middlewares.

Different behavior means a different model

When two entities start behaving differently, you create two models. Not a flag, not a type column, not a branching handler — a new model with its own operations. Your schema reflects your domain, not your edge cases.

Operations are explicit contracts

Every create, read, update, delete declares exactly what data it receives and what it can do with it. No implicit context, no hidden globals — the hook signature is the documentation.

Complex logic has a designated place

When business logic is too complex for the schema, it goes to an external worker — a plain HTTP service in any language. The boundary between schema and code is always clear and intentional.

Validation is structural, not procedural

Field constraints (required, positive, min) and predicate guards live on the model itself. Validation doesn't live in a separate layer that can drift out of sync.

Time is part of the model

Cron jobs are schema-level constructs defined next to the models they operate on. A billing cycle, a nightly cleanup, a stats refresh — they belong in the schema, not in a separate job runner configured elsewhere.

FQL

A language designed
for data modeling

FQL is not a config format. It's a small, purposeful language that describes how your data behaves.

Typed fields with validators

Fields carry their own constraints. required, min, max, pattern, positive — validated before any hook runs.

Scoped hook parameters

before(body, headers) declares exactly what's available. No global state, no implicit injection — the signature is the documentation.

Predicate guards with messages

body.age >= 18 : "Must be adult" — conditions that throw with your exact message. Guard clauses instead of try/catch.

Filter-based reads in hooks

Query related data inline: for acc in read Account { person_id: eq(body.id) }. Conditional logic via filtering, not branching.

cron + external example 
external PaymentGateway {
  input  { amount: currency  card_token: string }
  output { charge_id: string   success: boolean }
  url: "http://payment-worker:3001"
}

model Subscription {
  fields {
    customer_id: id       required
    card_token:  string   required
    amount:      currency
    charge_id:   string
    active:      boolean
  }
  read {}  create {}  update {}
}

cron {
  billing_cycle("0 0 * * *") {
    for sub in read Subscription { active: eq(true) } {
      res = PaymentGateway(amount: sub.amount  card_token: sub.card_token)
      update Subscription(sub.id) { charge_id = res.charge_id }
    }
  }
}
