model Order {
  fields {
    customer_id: id       required
    amount:      currency  positive
    status:      OrderStatus
  }

  create {
    before(body) {
      body.amount > 0           : "Amount must be positive"
      body.amount <= 10000      : "Amount exceeds limit"
    }
    after(body) {
      Notify(customer_id: body.customer_id  event: "order_created")
    }
  }

  read {}
  update {}
  delete {}
}
Philosophy

Built around one idea

Most backends are the same thing written differently. Fookie makes the common case trivial — and keeps the hard case possible.

📐

Schema is truth

Your FQL schema is the single source of truth. The database tables, GraphQL types, input validation, and API surface all derive from it automatically. Change the schema, everything updates.

🔁

Lifecycle over middleware

Every operation has a before and after hook with typed scope. No middleware chains, no magic injection — just explicit, readable hooks at the model level.

🚫

No branching in schema

FQL has no if. If behavior differs, models differ. This forces clean domain modeling instead of tangled conditionals hiding inside CRUD handlers.

🔌

Escape hatch built-in

Complex logic belongs outside the schema. external blocks let you call any HTTP worker — Node.js, Python, anything — and compose the result back into your FQL operations cleanly.

⚡

Zero-config GraphQL

Define a model, get create_x, all_x, update_x, delete_x, count_x mutations and queries immediately. No resolver code, no schema stitching.

🕐

Cron as first-class citizen

Background jobs are schema-level constructs, not afterthoughts. Write them in FQL next to the models they operate on, with full access to the same lifecycle and filtering primitives.

FQL

A language designed
for data modeling

FQL is not a config format. It's a small, purposeful language that describes how your data behaves.

Typed fields with validators

Fields carry their own constraints. required, min, max, pattern, positive — validated before any hook runs.

Scoped hook parameters

before(body, headers) declares exactly what's available. No global state, no implicit injection — the signature is the documentation.

Predicate guards with messages

body.age >= 18 : "Must be adult" — conditions that throw with your exact message. Guard clauses instead of try/catch.

Filter-based reads in hooks

Query related data inline: for acc in read Account { person_id: eq(body.id) }. Conditional logic via filtering, not branching.

cron + external example 
external PaymentGateway {
  body   { amount: currency  card_token: string }
  output { charge_id: string   success: boolean }
  url: "http://payment-worker:3001"
}

model Subscription {
  fields {
    customer_id: id       required
    card_token:  string   required
    amount:      currency
    charge_id:   string
    active:      boolean
  }
  read {}  create {}  update {}
}

cron {
  billing_cycle("0 0 * * *") {
    for sub in read Subscription { active: eq(true) } {
      res = PaymentGateway(amount: sub.amount  card_token: sub.card_token)
      update Subscription(sub.id) { charge_id = res.charge_id }
    }
  }
}
